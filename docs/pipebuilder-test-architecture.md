Status: implemented baseline

Scope: offline acceptance of `pipebuilder.py`, compatibility with four Agent clients, real-Agent acceptance, and release gates

## 1. Conclusion

All PipeBuilder test code uses Python. The project builds only E2E integration tests and does not maintain a unit-test layer split by internal function, class, or adapter.

Every test must:

1. Create a complete, isolated PipeSpace sandbox.
2. Use `subprocess` to execute the same release-candidate `pipebuilder.py`.
3. Accept results through externally observable behavior: CLI exit code, stdout/stderr, filesystem state, locks, and Agent-client behavior.
4. Never import `pipebuilder.py` or call an internal parser, resolver, renderer, or adapter.
5. Never reuse production normalization, selection, merge, digest, or rendering implementations to calculate expected values.

Test code may be split across multiple `.py` files for maintainability. The "single-file" constraint applies only to the release artifact:

```text
Release: pipebuilder.py
Tests: tests/e2e/**/*.py
```

JSON remains the format for PipeSpace inputs, CLI reports, and golden data. MJS, TypeScript, npm test runners, and shell test scripts are not used.

---

## 2. Why E2E-Only

PipeBuilder's primary risks do not concern whether an isolated pure function behaves according to its implementation. They concern boundary behavior across a complete build:

- Whether discovery, priority, explicit lists, and tag matching work together across multiple Skill Providers.
- Whether workspace paths are interpreted correctly under different working directories and operating systems.
- Whether each corresponding Agent can actually discover the projection generated by its adapter.
- Whether Human-owned sources remain byte-for-byte unchanged at all times.
- Whether managed artifacts, locks, and provenance remain consistent.
- Whether a failed build leaves only Builder-owned targets that a subsequent build can overwrite.
- Whether clean removes only content owned by PipeBuilder.
- Whether argv handling, paths, symlinks, secrets, or configuration merges cross a security boundary.

These properties can be established only through the complete CLI, a real filesystem, and real subprocess execution.

When AI generates both implementation and tests, fine-grained unit tests can also duplicate the implementation's incorrect assumptions: a production helper and a test helper may be wrong in exactly the same way. This project therefore bases its release gates on independent contracts, fixed inputs, human-reviewable goldens, negative scenarios, and real-client behavior.

This is neither a conceptual argument that Python's `unittest` module cannot be used for E2E testing nor a claim that unit tests have no value in other projects. It is an explicit engineering tradeoff for PipeBuilder: no unit suite is maintained, and the entire test budget is directed toward externally observable contracts.

## 3. System Under Test and Black-Box Boundary

### 3.1 Sole Entry Point Under Test

The runner must receive an explicit release-candidate path and launch it with the current Python interpreter:

```python
subprocess.run(
    [sys.executable, str(pipebuilder), "build", "--format", "json"],
    cwd=space_root,
    env=controlled_env,
    shell=False,
    text=True,
    capture_output=True,
)
```

The following is prohibited:

```python
import pipebuilder
pipebuilder.build(...)
```

Copying a production function into the tests and using it as an oracle is also prohibited.

### 3.2 Complete System Boundary

An offline E2E run includes at least:

```text
pipespace.json
<name>.code-workspace
PipeSpace .pipebuilder/agents/
PipeSpace .pipebuilder/skills/
one or more external folder Skill Providers
project directories referenced by the workspace
pipebuilder.py subprocess
real temporary filesystem
generated output from all four adapters
.pipebuilder/lock.json
.pipebuilder/build.lock
CLI JSON report
```

When an Agent CLI is installed, the test must pass the entire generated PipeSpace to the real client for inspection rather than merely asserting that files exist.

### 3.3 Non-Mocked Boundary

Offline E2E tests do not mock:

- File I/O, rename operations, permissions, or symlinks.
- The `pipebuilder.py` process.
- Final parsing of JSON, TOML, or frontmatter.
- Locks, concurrency locks, per-file atomic replacement, or clean.

Agent CLIs, accounts, and network access are not hard dependencies of offline PR tests. They belong to the next two E2E execution tiers.

## 4. Three E2E Execution Tiers

These tiers represent external dependencies and acceptance strength, not unit/integration code layering.

### E0: Offline PipeSpace E2E

Each case starts from complete PipeSpace input, executes the final `pipebuilder.py`, and inspects the complete post-build state.

Characteristics:

- No network access.
- No account.
- Does not require Codex, Cursor, CodeBuddy, or Claude Code to be installed.
- Runs on the native filesystem on Linux, Windows, and macOS.
- Highly deterministic and required for every PR.
- Covers most manifest, selection, adapter-projection, ownership, concurrency, and security contracts.

### E1: Installed-Client E2E

Uses the real PipeSpace produced by E0, then invokes an installed Agent CLI or official inspection command to verify that the client can parse and discover the generated output.

Characteristics:

- Uses a real client version.
- Prefers parse, check, policy, configuration-discovery, or hook-probe commands that do not initiate a model request.
- Records the Agent CLI name, version, platform, and actual command.
- Runs on nightly, main, or release runners.
- Under `--require`, a missing client is a failure rather than a skip.

The automated Codex E1 implementation covers native prompt assembly for generated `AGENTS.md`/Skills, trusted-project configuration discovery, project-hook schema parsing, and `codex execpolicy check`. Actual hook execution belongs to E2. Other platforms are added only after a stable official entry point has been confirmed for the relevant version; CLI arguments are never invented.

### E2: Real-Agent E2E

Starts a minimal real-Agent session in the generated sandbox and uses a sentinel Skill, rule, hook, command, or subagent to verify that the Agent actually consumed the artifact.

Characteristics:

- Requires an installed client, account, credentials, and network access.
- Opt-in by default.
- Does not compare complete natural-language output.
- Accepts only machine-verifiable sentinels, files, or structured events.
- Used for nightly runs, release candidates, or manual compatibility certification.
- Reported separately from the deterministic E0 gate so that model variability does not block routine development.

The initial E2 implementation uses Codex only. The current sentinel Skill is first committed to a disposable local Git repository, built through a Git Provider with branch/subdir selection, commit locking, and caching, and then consumed by a real Codex model. The same request also validates the generated `AGENTS.md` and the SessionStart hook. By default, the runner uses the installed Codex client's effective default model; a release job may pin a model explicitly with `--model`. A model ID that may be retired is never hard-coded in the repository. This setting affects E2 only: it does not enter E0/E1 or the PipeSpace product protocol. Cursor has completed manual real-client E1 certification, but no automated Cursor E1 case has yet been committed. CodeBuddy and Claude Code currently have E0 projection coverage only and remain `generated-only`; they will enter E1/E2 only after stable official entry points are confirmed.

        │   ├── offline/
        │   │   ├── test_contract.py
        │   │   ├── test_manifest_workspace.py
        │   │   ├── test_providers_skills.py
        │   │   ├── test_adapters.py
        │   │   └── test_lifecycle_security.py
        │   ├── client/

During initial development in the THarness repository, this directory may live at `harness/pipebuilder/`. It is moved unchanged when PipeBuilder is published independently.

The static all-agent fixture provides human-reviewable goldens. See `tests/e2e/COVERAGE.md` for the implemented requirement-to-case mapping. The [Skill Fixture Catalog](pipebuilder-skill-fixture-catalog.md) remains the target catalog for future expansion of the concrete fixture pack.

## 6. E2E Case Protocol

### 6.1 Directory for One Case

```text
minimal-all-agents/
├── case.py
├── input/
│   ├── space/
│   │   ├── pipespace.json
│   │   ├── demo.code-workspace
│   │   └── .pipebuilder/
│   ├── providers/
│   │   ├── provider-a/
│   │   └── provider-b/
│   └── projects/
│       ├── project-a/
│       └── project-b/
└── expected/
    ├── files/
    ├── tree.json
    ├── report.json
    └── lock.json
```

Relative relationships within `input/` are the actual relationships exercised by the test. The runner copies the entire `input/` tree into one temporary root. It must not copy only `space/` and then reconstruct Provider paths, because doing so would bypass real workspace and Provider path resolution.

### 6.2 Why Case Behavior Is Written in Python

The project does not define a `case.json` test DSL. Complex lifecycles are inherently procedural: build, modify a source or generated target, build again, clean, and compare state before and after. Python expresses this directly and avoids maintaining a separate interpreter solely for tests.

`pipespace.json` and expected JSON files are product inputs and acceptance data, not the test execution language.

### 6.3 Minimal `case.py`

```python
from support.case import PipeBuilderE2ECase


class Case(PipeBuilderE2ECase):
    id = "offline.build.minimal_all_agents"
    tier = "offline"
    tags = {"build", "all-agents", "golden"}

    def run(self):
        box = self.create_sandbox()
        inputs_before = box.snapshot_inputs()

        first = box.pipebuilder("build", "--format", "json")
        self.expect_exit(first, 0)
        self.expect_report_schema(first, "pipebuilder-report.v1")
        self.expect_golden_tree(box)
        self.expect_golden_lock(box)
        self.expect_inputs_unchanged(box, inputs_before)

        built = box.snapshot_all()
        second = box.pipebuilder("build", "--format", "json")
        self.expect_exit(second, 0)
        self.expect_snapshot_equal(box.snapshot_all(), built)
```

This test uses only test-side subprocess, snapshot, and assertion helpers. It does not access production internals.

### 6.4 metadata

Each case declares at least:

- A globally unique `id`.
- A `tier`: `offline`, `client`, or `live`.
- `tags`.
- Supported platforms.
- Corresponding requirement/diagnostic IDs.
- Whether parallel execution is allowed.
- Client capabilities required by a client/live case.

At startup, the runner checks for duplicate case IDs and complete metadata.

## 7. Sandbox and Environment Isolation

Each case creates a new `tempfile.TemporaryDirectory` with at least the following structure:

```text
<temp>/
├── input-copy/
│   ├── space/
│   ├── providers/
│   └── projects/
├── home/
├── tmp/
└── captures/
```

The runner constructs a minimal environment:

- An isolated `HOME`.
- Isolated `USERPROFILE`, `APPDATA`, and `LOCALAPPDATA` on Windows.
- An isolated temporary directory.
- Fixed locale and timezone plus controlled time input.
- User-level environment variables that could change Agent or PipeBuilder behavior are cleared.
- Only the minimal `PATH` required to launch Python and the client under test is retained.
- No `.codex`, Cursor, CodeBuddy, or Claude Code configuration is inherited from the real user.

If a client E2E case must read authentication state, it explicitly mounts a read-only credential location. The real home directory must never be used directly as the PipeSpace home.

Every subprocess is launched with an argv list and `shell=False`. The tests themselves must not depend on shell quoting for correctness.

## 8. Baseline Assertions for Every Offline E2E

Successful build cases check the following invariants by default. A case may disable one only for an explicit reason.

### 8.1 Input Immutability

The following inputs must remain byte-for-byte unchanged before and after a build:

- `pipespace.json`.
- `<name>.code-workspace`.
- `.pipebuilder/agents/` and `.pipebuilder/skills/`.
- Every external Skill Provider.
- Projects referenced by the workspace.

Platform target configuration is not Human-owned, including root `AGENTS.md`, `CLAUDE.md`, and `.mcp.json`. Tests must prove that these targets are determined entirely by `.pipebuilder/agents`, Skill `.pipe-agents`, and the core-generated workspace rule. After a target is modified directly, a subsequent build must restore the source-defined state.

### 8.2 Exact Generated Tree

Tests compare more than the existence of a few files:

- Path.
- File/directory/symlink kind.
- UTF-8 content or binary digest.
- Executable bit.
- Symlink target.
- Unexpected extra files.

In particular, tests must assert that `.pipe-agents` input does not leak into Agent-visible common Skill packages.

### 8.3 Report and Diagnostics

Using `--format json`, tests validate:

- The schema is `pipebuilder-report.v1`.
- CLI command name, status, and exit semantics.
- Stable diagnostic codes.
- Operation summary.
- Selected and shadowed Skills.
- Provenance.
- Build-lock state.
- Absence of credentials and environment secrets from the report.

Tests depend on stable codes and structured fields; they do not compare complete, mutable human-facing messages.

### 8.4 Lock and Provenance

Tests check that:

- Every managed artifact has an owner, source, digest, and adapter.
- The lock does not record machine-specific, non-portable paths unless the protocol explicitly requires and normalizes them.
- Source digests match the actual fixture content.
- Shadowed providers and explicit-selection reasons are explainable.
- The lock contains no tokens, complete sensitive environment variables, or secret values.

### 8.5 Idempotency

After the second build:

- The generated tree is unchanged.
- Human inputs are unchanged.
- The semantic content of the lock is unchanged.
- Only protocol-declared volatile report fields, such as timestamps and duration, may change.
- No temporary files or residual `build.lock` remain.

### 8.6 Clean as the Controlled Inverse of Build

After clean:

- Only artifacts owned by PipeBuilder in the lock are removed.
- Human-owned files and files subsequently added by the user remain unchanged.
- Nonempty human-created directories are not removed recursively.
- The PipeSpace can be built again with the same result.
- Repeated clean operations have explicit, stable idempotent semantics.

### 8.7 Simplified Failure Semantics

When an input conflict, invalid manifest, adapter conflict, or pre-write validation failure occurs:

- The final tree remains in its pre-invocation state.
- The lock does not advance to the new state.
- Temporary files are not exposed as final artifacts.

The apply phase guarantees per-file atomic replacement only, not a transaction across files. Tests must verify that the old `lock.json` remains unchanged after an interrupted apply and that, after the stale `build.lock` is removed, a subsequent build converges from source to a complete result. Transaction journals and automatic rollback are not tested.

## 9. Golden Strategy

### 9.1 Golden Contents

Stable successful scenarios preserve:

- `expected/files/`: full contents of key files that should be generated.
- `expected/tree.json`: the complete managed tree and digests.
- `expected/report.json`: the normalized structured report.
- `expected/lock.json`: the normalized lock.

Not every negative case requires complete file goldens, but every such case must define an exit code, diagnostic code, side-effect-free snapshot, and any necessary report golden.

### 9.2 Dynamic-Field Normalization

Only allowlisted substitutions are permitted:

```text
$SPACE_ROOT
$SANDBOX_ROOT
$BUILDER_SHA256
$FIXED_TIME
$PLATFORM
```

The normalizer resides on the test side, and its rules must be simple, explicit, and reviewable. It must not sort arrays with meaningful order, delete unknown fields, or generalize every absolute path or timestamp in a way that conceals errors.

### 9.3 Update Process

Goldens may be updated only explicitly:

```bash
python3 tests/e2e/tools/update_goldens.py \
  --case offline.build.minimal_all_agents
```

CI never updates goldens automatically. An update must review:

- Why the contract changed.
- The tree diff.
- The report/lock diff.
- The projection diff for all four Agents.
- Whether a local machine path or secret was accidentally written into expected data.

Test helpers must not call a production renderer to generate expected files. A golden updater may capture actual output once, but that capture is only a review candidate, not an automatically correct oracle.

## 10. Required E2E Scenario Matrix

### 10.1 Manifest and Workspace

- Minimal four-Agent PipeSpace.
- `pipespace.json.name` differs from the directory basename.
- Missing, multiple, or name-mismatched workspace files.
- Malformed JSON, unsupported schema/version, and unknown keys.
- When only the legacy THarness `tagents/private/.harness-space.yaml/.harness-lock.yaml` layout exists, return `PB015` with zero writes.
- When legacy and current layouts coexist, do not merge them or infer priority.
- A workspace with one folder using `path: "."`, an external relative folder, a mixture of both forms, and multiple folders.
- Absolute, duplicate, and missing workspace folder paths.
- Paths containing spaces, Unicode, `#`, quotes, and shell metacharacters.
- Building a PipeSpace from its own cwd and from an explicit directory argument.
- `check`, `explain`, and `build --dry-run` produce no writes.

### 10.2 Skill Providers and Selection

- Zero, one, and multiple folder Providers.
- Colocated/external-relative/symlink roots for folder Providers, content-digest updates, missing/non-directory roots, generated-target recursion, and symlink loops.
- The Git Provider's mutually exclusive `branch`/`tag` schema, subdir, and rejection of credentials in URLs.
- Online advancement of Git branches, fixed tags, and lock provenance from branch/tag to immutable commit.
- PipeSpace-local Git cache, no mutable checkout, reuse of the locked commit under `--offline`, missing cache, and snapshot-digest tampering.
- Git archive symlinks, missing branch/tag/subdir, and stable diagnostics for Git-command failures.
- Strict shadowing by the unified Provider order when folder, Git, and space-local providers are mixed.
- Highest priority for PipeSpace-local `.pipebuilder/skills`.
- Priority of the explicit `skills` list.
- Union of tag matches.
- Same-name Skill shadowing and provenance.
- Changes in provider order.
- Missing `SKILL.md`, invalid Skill names, and duplicate canonical names.
- Provider path escape, symlink escape, and loops.
- Unselected Skills do not enter any Agent output.

### 10.3 Four Adapters

Each adapter is covered by complete PipeSpace build cases:

- Installation of portable common Skills and full-capability Skills for the platform.
- Workspace rule.
- Every supported native source surface for the platform.
- Explicit rejection of gated or unsupported surfaces.
- Merging Space-level and Skill-level artifacts.
- Unknown directories.
- Semantic-key conflicts.
- Human-owned source immutability and regeneration of Builder-owned targets.
- Semantic configuration merge.
- Hook/MCP secret linting.
- Clean and repeated build.
- Windows path normalization.

Additional Codex coverage:

- Stable composition of the canonical workspace rule, Space `AGENTS.md` source, and Skill `AGENTS.md` source.
- Native discovery of the generated root `AGENTS.md` and recovery from target drift.
- Project configuration discovery.
- `.codex/rules` command policy.
- `.codex/hooks.json` source merge and project-trust behavior.

All of these are tested through complete builds and final file/client behavior. Directly constructing an adapter class and calling render is prohibited.

### 10.4 Ownership and Lifecycle

- Initial build.
- Repeated build with no changes.
- Controlled update after a source change.
- Rebuild restores the source-defined state after direct modification of a generated target.
- Sibling files unrelated to the plan remain unchanged.
- Missing or corrupt lock, old lock schema, and Builder version changes.
- Deletion or type change of a managed file.
- Clean removes only owned artifacts.
- Build/clean concurrency-lock contention.
- Recovery through a subsequent build after failure.

### 10.5 Simplified Writes and Concurrency

- Zero writes when planning fails.
- Each target file is written through a same-directory temporary file followed by atomic replacement.
- No temporary files remain after success.
- Return `PB013` when `build.lock` already exists.
- Return `PB014` for a stale `build.lock`, which is deleted only after Human confirmation.
- When apply is interrupted, the old `lock.json` does not advance; a subsequent build converges completely.
- No transaction journal is generated.

### 10.6 Security and Portability

- `..`, absolute targets, and symlink escape.
- Symlink loops.
- Windows junctions/reparse points.
- Windows drives, UNC paths, and case-insensitive collisions.
- Case-sensitivity differences between macOS and Linux.
- Reserved device names and invalid filenames.
- Linting/redaction when secrets appear in a manifest, hook, or MCP configuration.
- Executable bit.
- UTF-8 BOM, CRLF/LF, and Unicode normalization.
- Excessively long paths and deeply nested Skill trees.

Platform filesystem behavior must be accepted on a native runner for the corresponding OS; string simulation of Windows or macOS is insufficient.

### 10.7 Diagnostics

Every stable diagnostic code defined by the PipeSpace protocol must map to at least one complete failing E2E case. `COVERAGE.md` maps requirements and diagnostics to case IDs. A diagnostic code without an E2E case cannot enter the stable contract.

## 11. Real-Client E2E

### 11.1 Two-Stage Execution

Every E1 case performs:

```text
Stage A: pipebuilder.py builds the complete sandbox
Stage B: the real Agent client reads the same sandbox
```

If Stage A fails, Stage B does not run. If Stage B fails, the complete build tree and client output are retained.

### 11.2 Capability probe

Tests must not infer capabilities solely from an executable filename. The runner first records:

- Executable realpath.
- Version.
- OS/architecture.
- Supported network-free verification entry points.
- Whether an account is available and whether the case uses it.
- Verification level.

Recommended verification levels in the report:

```text
generated-only
client-parsed
client-discovered
live-consumed
```

### 11.3 Permitted Skip Boundary

- During an ordinary development E1 run, an uninstalled client may produce a structured skip.
- Nightly execution should include at least one fixed runner for each platform.
- Release execution with `--require --agent <id>` does not permit skips.
- "Command not found" must be distinguished from "command executed but incompatible."
- A version outside the support matrix is reported as unsupported, never as successful.

### 11.4 No Guessing About Platform Behavior

Each platform's client cases must reference official capabilities and commands confirmed in the adapter documentation. When a client upgrade changes an entry point, update the compatibility note and E1 case before claiming support for the new version.

## 12. Real-Agent E2E

### 12.1 Codex Default-Model Profile

Codex E2 does not hard-code a default model ID in the test repository. Ordinary live runs use the installed Codex client's currently effective default model. When release reproducibility is required, the release job explicitly passes `--model <model-id>`, and `e2e-report.json` records the requested override, Codex client version, actual argv, and result.

The runner's model-selection precedence is:

```text
CLI --model
> installed Codex effective default
```

This prevents an obsolete model name in documentation from making the suite unusable, while also preventing environment variables from silently changing the release profile. If client events do not expose the effective model, the report may state only "using client default"; it must not guess a specific ID.

The runner's final invocation must be equivalent to:

```text
codex --ask-for-approval never \
  --sandbox workspace-write \
  --cd <space-sandbox> \
  --dangerously-bypass-hook-trust \
  exec \
  --ephemeral \
  --json \
  --output-schema <schema> \
  --output-last-message <capture> \
  <sentinel-prompt>
```

The Python runner still launches this command with an argv list and `shell=False`. It must not use `--dangerously-bypass-approvals-and-sandbox`. The current combined sentinel must validate a project hook, so `--dangerously-bypass-hook-trust` is used only in a disposable sandbox where the case itself constructs both the hook source and path. The temporary `CODEX_HOME` mounts existing authentication only through a symlink; credentials are not copied into fixtures or reports.

### 12.2 Sentinel Acceptance

Complete real-model output is unsuitable for a golden. Each E2 case uses a unique sentinel that cannot match accidentally, for example:

```text
HARNESSBUILDER_E2E_SKILL_7F31
```

Acceptable verification methods include:

- A designated Skill creates a sandbox file with constrained content after it is triggered.
- A hook writes the sentinel to a capture file.
- A subagent returns a structured sentinel.
- A command receives the expected arguments and leaves a receipt.
- A rule produces a machine-verifiable result for an explicit allow/deny command.

The prompt must be short, deterministic, and free of business data. Tests must not allow the Agent to access files outside the sandbox, modify the repository, or use production credentials. Each execution sets time and token limits and records cost, model, and client version as report metadata.

E2 failures inform compatibility warnings and release review. Whether they block a release depends on the platform-stability policy, but retrying until success must not conceal the actual failure rate. When a fixed number of retries is permitted, every result must be reported.

## 13. Python Runner

### 13.1 Commands

```bash
python3 tests/e2e/run.py --tier offline
python3 tests/e2e/run.py --tier offline --case offline.concurrency.build_busy
python3 tests/e2e/run.py --tier client --agent codex --require
python3 tests/e2e/run.py --tier live --agent codex --require
python3 tests/e2e/run.py --tier live --agent codex --model <model-id> --require
python3 tests/e2e/run.py --tier offline --jobs 8
```

The runner uses only the Python standard library. It currently uses `ThreadPoolExecutor` to run subprocess-heavy, parallel-safe cases concurrently. Every case still has an independent sandbox; concurrency-lock and real-client cases marked serial run separately.

### 13.2 Runner Output

Human-readable output remains concise:

```text
PASS offline.build.minimal_all_agents  0.42s
FAIL offline.ownership.generated_rebuild  0.11s
SKIP client.cursor.discovery            client-not-installed
```

The runner also generates a structured `e2e-report.json` containing at least:

- Suite schema/version.
- Release-artifact path and SHA256.
- Python/OS information.
- Case ID, tier, duration, and status.
- PipeBuilder exit/report.
- Agent client version, requested/effective model, and override source.
- Skip/failure reason.
- Artifact-capture path.

The release report must prove that every case executed the same SHA256 of `pipebuilder.py`.

### 13.3 Failure Retention

On failure, the sandbox is copied to:

```text
tests/e2e/.artifacts/<run-id>/<case-id>/
```

It includes:

- Input copy.
- Actual output tree.
- Normalized and unnormalized reports.
- stdout/stderr.
- Snapshot diff.
- `build.lock` and temporary-file state.
- Client capability/version.
- A directly reproducible description of argv and cwd.

Secrets must not be retained; redaction runs again before capture.

## 14. CI and Release Gates

### 14.1 Pull Request

The following must pass:

- All E0 offline cases.
- Native Linux and Windows runners.
- Review of changed goldens.
- Execution of the single-file release candidate from a temporary location, proving that it does not depend on the repository import path.

A full macOS E0 run is recommended on main or nightly. Required cases related to the macOS filesystem must run before release.

### 14.2 Main/nightly

- Full E0 on Linux, Windows, and macOS.
- E1 on fixed runners for all four platforms.
- Crash/kill, concurrency, and large-fixture stress cases.
- Optional Codex E2 live sentinel using the client default or an explicit release-job `--model`.
- Client-version changes and compatibility-matrix reports.

### 14.3 Release candidate

1. Freeze one `pipebuilder.py` SHA256.
2. Run E0 on all three operating systems against that content-identical artifact.
3. Run E1 for all four Agents with `--require`; silent skips are prohibited.
4. Report E2 results separately.
5. Archive the structured report, client versions, and checksum.
6. The published artifact after testing must still have the same SHA256.

## 15. Preventing AI-Generated Tests from Proving Themselves Correct

E2E tests are not automatically reliable; oracle independence must be maintained deliberately:

1. Behavioral expectations come from a versioned protocol, not the current implementation structure.
2. Tests must not import production code.
3. Test helpers do not implement a Provider resolver or adapter renderer.
4. Goldens are committed as explicit files and structured results, then reviewed by a Human.
5. Every happy path has at least one corresponding input mutation or failure scenario.
6. Assertions cover the complete tree, input immutability, lock/provenance, and repeated builds, not only the exit code.
7. Critical adapters are parsed or consumed by real clients.
8. Positive and negative cases for the same requirement use different fixtures.
9. When fixing a bug, first commit a complete reproduction case that fails against the old artifact.
10. Broad golden changes must explain the contract change; they cannot be accepted with a simple "accept all" action.

Here, mutation means externally changing the state of a PipeSpace, Provider, generated artifact, lock, or build lock and then rerunning the complete CLI. It does not mean calling an internal function in a miniature test.

## 16. Migration from Existing Builder Tests

Tests for the current Node/TypeScript Builder will not be expanded further and will be migrated gradually to Python E2E:

| Existing content | New location | Migration approach |
| --- | --- | --- |
| `tests/integration/builtin-builder.mjs` | `cases/offline/build/cursor_legacy_parity/` | Convert to a complete `pipespace.json`, workspace, Provider, and Cursor artifact goldens |
| `tests/fixtures/builtin/*` | `cases/offline/**/input/` | Convert to standard `skills/foo/SKILL.md` and `.pipe-agents/` |

Existing command-runner tests are not migrated into PipeBuilder; the build command has been removed from the new protocol.

During migration, the legacy Builder's own Node integration tests may remain temporarily, but every new PipeBuilder behavior must enter the Python E2E suite. After PipeBuilder reaches Cursor parity, the legacy suite will be retired with the legacy Builder as tests of the old implementation.

## 17. Minimum Initial Implementation Order

### Step 1: Establish the Runner and Three Baseline Cases

- Minimal all-agents build.
- Malformed manifest with zero writes.
- The build -> build -> clean -> build lifecycle.

These three cases first establish the sandbox, snapshot, report, and golden protocols.

### Step 2: Provider/Selection and Four Adapters

- Provider priority.
- Explicit Skill + tag union.
- Complete successful projection for each platform.
- Semantic conflicts, Human-owned source immutability, and regeneration of generated targets.

### Step 3: Ownership/Concurrency/Security

- Managed-artifact mutation.
- Stale/corrupt lock.
- Busy/stale `build.lock`.
- Symlink/path escape.

### Step 4: Real Clients

- Integrate Codex E1 first.
- For each platform, establish a fixed runner and capability report after confirming a stable entry point.
- Add the minimal E2 sentinel last.

The initial release cannot implement only the happy path. It is eligible to replace the existing Builder only after the complete E0 matrix, all four adapter projections, ownership, clean, concurrency, and security boundaries pass.

## 18. Acceptance Criteria

The test architecture is complete when:

1. All test execution code is Python.
2. There is no PipeBuilder unit suite and no adapter direct-call test.
3. Every case executes the release-candidate `pipebuilder.py` through a subprocess.
4. A PR can complete all E0 tests without an Agent, account, or network access.
5. All four Agents have complete PipeSpace projection cases and real-client E1 entry points.
6. Build, repeated build, convergence after an interrupted apply, clean, and build-again all have E2E coverage.
7. Human-owned inputs, ownership, provenance, secret redaction, and path boundaries have negative validation.
8. Platform behavior on Linux, Windows, and macOS is validated on native runners.
9. The release report traces to one unique `pipebuilder.py` SHA256.
10. Every stable diagnostic and supported adapter capability maps to at least one E2E case.
