# PipeBuilder Initial Four-Agent Adapter Specification

Status: implemented
Date: 2026-07-13
Target platforms: Codex, Cursor, CodeBuddy, and Claude Code

This document defines the input structure, target mappings, workspace-rule projections, merge boundaries, and future plugin evolution for the initial adapters. Platform behavior may change across product versions, so implementations must be validated with fixtures against real clients. Any schema that cannot be confirmed must fail closed rather than be inferred.

Current compatibility status: Codex has automated E1 verification against a real client, while Cursor has completed manual E1 verification against a real client; both are marked `client-verified`. CodeBuddy and Claude Code have E0 generation plus the limited structural validation supported by the current implementation, and remain marked `generated-only` until their respective E1 verification is established. Cursor's manual E1 has not yet been codified as an automated case and must not be represented as equivalent to Codex's continuous verification level. This status is recorded in `explain` and `.pipebuilder/lock.json`; it does not mean that the adapter has been removed.

---

## 1. Common Inputs

Each selected Skill provides:

```text
<skill>/                         common package
<skill>/.pipe-agents/<agent>/      agent-specific artifacts
```

The PipeSpace provides:

```text
.pipebuilder/agents/<agent>/      Space-level Agent-specific inputs
```

The adapter consumes the IR resolved by the core and does not traverse Providers itself:

```text
HarnessSpaceContext
WorkspaceRule
SelectedSkills[]
AgentArtifacts[]
PreviousOwnership
```

The adapter emits build operations and does not write directly to disk:

```text
copy
render
merge-document
```

The core centrally enforces conflict handling, security checks, ownership, and per-file atomic writes.

---

## 2. Agent-Native Source Layouts

Both the Space-level `.pipebuilder/agents/<agent>/` tree and the Skill-level `.pipe-agents/<agent>/` tree are treated as virtual project roots for the corresponding Agent, preserving complete native target-relative paths. For example, Codex uses `AGENTS.md`, `.codex/config.toml`, `.codex/hooks.json`, `.codex/hooks/`, and `.codex/rules/`, while Claude Code uses `CLAUDE.md`, `.claude/settings.json`, `.claude/rules/`, `.claude/agents/`, `.claude/hooks/`, and `.mcp.json`. Generic intermediate directories such as `config/`, `mcp/`, and `files/` are no longer used.

Each platform adapter independently defines:

- supported directories;
- file schemas;
- semantic keys;
- target paths;
- merge policies;
- trust and security diagnostics.

Any unknown native file, unknown directory, or known but unimplemented surface must produce an error. When multiple Space or Skill sources feed the same target configuration file, the adapter first merges them by platform semantic key, after which the core generates the complete Builder-owned target.

## 3. Cross-Platform Overview

| Logical capability | Codex | Cursor | CodeBuddy | Claude Code |
| --- | --- | --- | --- | --- |
| Common Skill | `.agents/skills/<name>` | `.cursor/skills/<name>` | `.codebuddy/skills/<name>` | `.claude/skills/<name>` |
| Workspace rule | Generated section in the root `AGENTS.md` | `.cursor/rules/pipebuilder-workspace.mdc` | `.codebuddy/rules/pipebuilder-workspace.md`, fixture required | `.claude/rules/pipebuilder-workspace.md` |
| Rules | `.codex/rules/**/*.rules` (command policy, not workspace guidance) | `.cursor/rules/**/*.mdc` | Fixed workspace rule only; arbitrary native Rules from capability packages are currently gated | `.claude/rules/**/*.md` |
| Commands | Prefer conversion to a Skill; legacy commands are not generated by default | `.cursor/commands/**/*.md` | `.codebuddy/commands/**/*.md` | `.claude/commands/**/*.md` compatibility |
| Agents | Agent roles in `.codex/config.toml` | Cursor agents/plugin, version-specific fixture required | `.codebuddy/agents/**/*.md` | `.claude/agents/**/*.md` |
| Hooks | `.codex/hooks.json` | Cursor hooks, version-specific fixture required | `.codebuddy/settings.json` | `.claude/settings.json` |
| MCP | `.codex/config.toml` | Cursor MCP target, version-specific fixture required | `.codebuddy/mcp.json` | `.mcp.json` |

"Fixture required" means that the platform supports the capability, but the target path and schema must be confirmed against the team's actual client version before the initial PipeBuilder implementation can enable it. Simple file copying is not a substitute for validation.

## 4. Codex Adapter

Agent ID:

```text
codex
```

### 4.1 Skill

```text
<skill common package>
  -> .agents/skills/<skill-name>/
```

The copy excludes `.pipe-agents/`. Unknown frontmatter in `SKILL.md` is preserved.

The Skill is not also copied to `.codex/skills`, which avoids duplicate discovery. If a real deployment still requires the legacy path, it must be an explicit adapter compatibility option rather than default behavior.

`.agents/skills` is the Codex Skill surface; `.codex` is the optional project config, hooks, and rules surface. The adapter does not create `.codex` unless an explicit `.pipebuilder/agents/codex/.codex` or `<skill>/.pipe-agents/codex/.codex` source exists.

### 4.2 Workspace Rule and `AGENTS.md`

The Codex adapter reads `.pipebuilder/generated/workspace-rule.md`, which is generated by the core, and combines it with native project instructions from the following Builder sources:

```text
<space-root>/.pipebuilder/agents/codex/AGENTS.md
<skill>/.pipe-agents/codex/AGENTS.md
```

The target is the root `AGENTS.md` in the PipeSpace, and the complete file is managed by the Builder. The composition order is fixed: the canonical workspace section, the Space source, and then Skill sources in final Skill installation order. Each section has a stable generated heading and source identity. Sections with identical body digests may be deduplicated; different sources do not override one another.

PipeBuilder does not generate a workspace-context hook, nor does it require Codex to read `.code-workspace` or `.pipebuilder/generated/` at runtime. Codex discovers the generated `AGENTS.md` through its native rule-discovery process from the project root to the current directory. Consequently, launching from the PipeSpace root directly exposes the folder inventory. Launching independently from a decoupled external folder is not the same PipeSpace session, and the Builder does not attempt to write targets outside the Space root.

### 4.3 Config, Agents, and MCP

Inputs:

```text
<space-root>/.pipebuilder/agents/codex/.codex/config.toml
<skill>/.pipe-agents/codex/.codex/config.toml
```

Target:

```text
.codex/config.toml
```

The file must be merged according to Codex configuration semantics:

- `[agents.<name>]` uses the agent name as its key;
- MCP uses the server name as its key;
- identical definitions under the same key may be deduplicated;
- different definitions under the same key fail;
- project-unsafe fields and credential fields must not be generated;
- the lock records the target, sources, and final digest.

Multiple sources are merged by Codex configuration semantic key. The target `.codex/config.toml` is managed in its entirety by the Builder; manually authored fields and comments in the target are neither read nor preserved. Human-authored configuration must be written back to `.pipebuilder/agents/codex/.codex/config.toml`. The initial version may support only a strict TOML subset verified by E1.

### 4.4 Rules

```text
<agent-source-root>/.codex/rules/**/*.rules
  -> .codex/rules/**/*.rules
```

Native Codex `.rules` files define command permission policies outside the sandbox; they are not Markdown project guidance. The adapter must:

- accept only the `.rules` extension;
- preserve relative paths;
- check target collisions and ownership;
- emit security diagnostics for broad `allow` policies, external script paths, and suspicious shell wrappers;
- use a fixture for the current Codex version and, when available, validate rule loading with `codex execpolicy check`;
- mark the capability as an experimental platform surface in the lock.

The project structure declared by `.code-workspace` is not written to `.codex/rules/`; it is provided only by the root `AGENTS.md` generated in Section 4.2.

### 4.5 Hooks

Input sources:

```text
<space-root>/.pipebuilder/agents/codex/.codex/hooks.json
<skill>/.pipe-agents/codex/.codex/hooks.json
```

These hooks are additive. The listing order above is used only for stable diagnostics and does not imply execution order or override precedence.

Target:

```text
.codex/hooks.json
```

Hooks use the current native three-level Codex structure: event -> matcher group -> `hooks` handlers. They are merged additively by event. Completely identical matcher groups are deduplicated, while handlers from different groups are not speculatively reordered or collapsed. Hook commands, network URLs, and external paths are included in the security report. The final schema is verified by the E1 real-Codex-client parse case.

No behavior may depend on the execution order of multiple hooks.

### 4.6 Commands

Reusable Codex workflows should be modeled as standard Skills. In the initial version, the presence of `commands/` in any Codex source is treated as unsupported to avoid continuing to generate a diminished or deprecated custom-prompt surface.

## 5. Cursor Adapter

Agent ID:

```text
cursor
```

### 5.1 Skill

```text
<skill common package>
  -> .cursor/skills/<skill-name>/
```

### 5.2 Workspace Rule

Target:

```text
.cursor/rules/pipebuilder-workspace.mdc
```

Use the following fixed frontmatter:

```yaml
---
description: PipeBuilder workspace folder inventory.
alwaysApply: true
---
```

The body is rendered from the WorkspaceRule IR.

### 5.3 Rules

```text
<agent-source-root>/.cursor/rules/**/*.mdc
  -> .cursor/rules/**/*.mdc
```

The adapter must validate:

- the `.mdc` extension;
- the presence of frontmatter;
- declaration of at least one of `description`, `alwaysApply`, or `globs`;
- target-path uniqueness;
- failure when the same path has different content.

The verified behavior of the current THarness Cursor adapter is preserved.

### 5.4 Commands

```text
<agent-source-root>/.cursor/commands/**/*.md
  -> .cursor/commands/**/*.md
```

Conflicts are detected not only by target path but also by the final slash-command name.

### 5.5 Hooks, Agents, Config, and MCP

Cursor supports these extension surfaces, but they evolve rapidly across versions. The initial-version policy is:

- enable only semantic documents covered by a real-client fixture;
- report unverified directories as unsupported;
- use native Cursor filenames and schemas without introducing generic config or MCP wrappers;
- semantically merge Builder sources that must be aggregated, with the entire target file managed by the Builder;
- record the adapter version in the lock.

## 6. CodeBuddy Adapter

Agent ID:

```text
codebuddy
```

### 6.1 Skill

```text
<skill common package>
  -> .codebuddy/skills/<skill-name>/
```

CodeBuddy-specific Skill frontmatter is preserved verbatim. PipeBuilder does not remove unknown fields on behalf of other Agents.

### 6.2 Workspace Rule

Preferred target:

```text
.codebuddy/rules/pipebuilder-workspace.md
```

Because the CodeBuddy IDE/CLI rule-discovery surface must be validated against the team's actual version, the adapter must change the target if the initial-version fixture proves that this path differs. The WorkspaceRule IR remains unchanged.

### 6.3 Commands

```text
<agent-source-root>/.codebuddy/commands/**/*.md
  -> .codebuddy/commands/**/*.md
```

Conflicts are detected by final slash-command name; the subdirectory namespace must be included in the semantic key.

### 6.4 Agents

```text
<agent-source-root>/.codebuddy/agents/**/*.md
  -> .codebuddy/agents/**/*.md
```

Validate frontmatter, agent name, mode, and tool fields. Different definitions under the same agent name fail.

### 6.5 Hooks/settings

Target:

```text
.codebuddy/settings.json
```

Inputs use the native `settings.json` in both scopes:

```text
<space-root>/.pipebuilder/agents/codebuddy/.codebuddy/settings.json
<skill>/.pipe-agents/codebuddy/.codebuddy/settings.json
```

Hooks are merged additively by event. Handlers for the same event may run concurrently, so ordering dependencies cannot be expressed. The target settings file is managed in its entirety by the Builder; manually authored fields in the target are not preserved.

Hooks execute code automatically, so the lock and `explain` must expose the command, matcher, source, and risk.

### 6.6 MCP

Target:

```text
.codebuddy/mcp.json
```

Inputs are the `.codebuddy/mcp.json` files in the two virtual project roots. Merge by server name. Only environment-variable references or external credential helpers are allowed; secret literals are prohibited.

## 7. Claude Code Adapter

Agent ID:

```text
claude-code
```

### 7.1 Skill

Claude Code currently uses standard Agent Skills:

```text
<skill common package>
  -> .claude/skills/<skill-name>/
```

The model may select a Skill automatically by its description, or the Skill may be invoked through `/skill-name`. Claude Code's legacy custom commands have been incorporated into the Skills mechanism, so new workflows should prefer Skills.

### 7.2 Workspace Rule

Target:

```text
.claude/rules/pipebuilder-workspace.md
```

No `paths` frontmatter is set, so the rule applies throughout the PipeSpace.

The root `CLAUDE.md` is also a Builder-owned target. When `CLAUDE.md` exists in either source scope, the source files are composed into the root file in stable source order. Human-authored content that must be preserved must reside in `.pipebuilder/agents/claude-code/CLAUDE.md`. The canonical workspace rule is still projected to `.claude/rules/pipebuilder-workspace.md` to avoid duplicating the same folder inventory across both targets.

### 7.3 Rules

```text
<agent-source-root>/.claude/rules/**/*.md
  -> .claude/rules/**/*.md
```

Always-loaded rules without frontmatter and path-scoped rules with a `paths` list are supported. The Builder validates the frontmatter shape and glob strings but does not rewrite the body.

### 7.4 Commands

Compatibility mapping:

```text
<agent-source-root>/.claude/commands/**/*.md
  -> .claude/commands/**/*.md
```

Claude Code remains compatible with this directory, but new capabilities should be authored as Skills. `explain` emits a migration warning for commands.

### 7.5 Agents

```text
<agent-source-root>/.claude/agents/**/*.md
  -> .claude/agents/**/*.md
```

At minimum, validate:

- `name`;
- `description`;
- agent-name uniqueness;
- the shape of `tools` and `disallowedTools`;
- that `skills` references exist;
- that `mcpServers` contains no secret literals;
- the hooks shape;
- enumerated fields such as `isolation: worktree`.

### 7.6 Hooks/settings

PipeSpace project settings:

```text
.claude/settings.json
```

Inputs are the `.claude/settings.json` files in the two virtual project roots. Hooks are a semantic section of settings. PipeBuilder:

- merges additively by event;
- normalizes and deduplicates entries;
- fails on conflicts;
- generates the complete Builder-owned `.claude/settings.json`;
- does not write `.claude/settings.local.json`;
- does not write managed or user settings;
- does not set high-risk defaults such as `bypassPermissions`.

### 7.7 MCP

Project-shared MCP target:

```text
.mcp.json
```

Both scopes use the native project path `.mcp.json` as their source and generate the project-shared `.mcp.json`. Sources are semantically merged by server name. The entire target is managed by the Builder; local and user scopes do not belong to the PipeSpace Builder.

### 7.8 Relationship to Plugins

Claude Code plugins can distribute Skills, agents, hooks, and MCP servers, but the initial PipeBuilder version does not compile a PipeSpace into a Claude plugin:

- a PipeSpace build is local to its task-specific agent pipeline root;
- a plugin is an installable, namespaced, versioned distribution unit;
- a Skill Provider may reference a plugin source in the future, but the two ownership models must remain distinct.

## 8. Cross-Platform Consistency of the Workspace Rule

The text generated for all four platforms must express the same facts:

```text
PipeSpace: <pipespace.json.name>
Workspace: <pipespace.json.name>.code-workspace
Folders, in declared order: <folders[] name/path>
```

They must also include the same rules:

1. Paths and project names may be obtained only from `.code-workspace`.
2. `path: "."` means that the project and the PipeSpace root share the same directory; other relative paths represent decoupled directories.
3. Folder order exists only for stable presentation and does not express primary/reference roles or permissions.
4. `.pipebuilder/agents/` and `.pipebuilder/skills/` are PipeBuilder sources, not generated directories.
5. A Skill's `.pipe-agents/` directory is a Skill source extension and is not copied into the common package.
6. Agents must not hand-edit `.pipebuilder/lock.json` or platform-generated targets.
7. Platform-rendered files are projections, not the programmatic source of truth.

Adapters may adjust frontmatter and platform terminology, but they must not change these facts.

## 9. Artifact Conflict Policy

| Artifact | Semantic key | Default policy |
| --- | --- | --- |
| Skill | Agent + Skill name | Exactly one copy after Provider resolution |
| Rule | Agent + logical rule ID/target | Deduplicate identical digests; otherwise fail |
| Command | Agent + slash-command name | Deduplicate identical digests; otherwise fail |
| Agent | Agent platform + subagent name | Deduplicate identical definitions; otherwise fail |
| Hook | Agent + event + normalized handler | additive + dedupe |
| MCP | Agent + server name | Deduplicate identical definitions; otherwise fail |
| Config | Agent + semantic key path | Explicit merge policy; otherwise fail |

A PipeSpace `.pipebuilder/agents` artifact does not automatically override a Skill `.pipe-agents` artifact merely because it has broader scope. Any future override protocol must explicitly declare the semantic key and the source being overridden; the initial version fails by default.

## 10. Adapter E2E Requirements

For the complete test directory, PipeSpace sandbox, goldens, and real-client protocol, see [PipeBuilder Python E2E Integration Test Architecture](pipebuilder-test-architecture.md). For standard Skill packs and the capability matrix, see [PipeBuilder Skill Fixture Catalog](pipebuilder-skill-fixture-catalog.md). This section defines only the minimum E2E coverage required for each adapter.

Each built-in adapter must have at least the following complete PipeSpace E2E cases:

- portable common Skills and one full-capability Skill for that platform;
- one workspace-rule fixture;
- one successful fixture for every supported native source surface;
- one failing fixture for every gated or unsupported native source surface;
- an unknown-directory failure fixture;
- a same-semantic-key conflict fixture;
- a fixture proving Human-owned source preservation and Builder-owned target regeneration;
- a repeated-build fixture;
- a clean fixture;
- a Windows path-normalization fixture;
- a platform-configuration merge fixture;
- a hook/MCP secret-lint fixture.

Codex coverage must also include source composition for the generated `AGENTS.md`, workspace inventory, target-drift recovery, the effect of project trust on `.codex/` surfaces, and a discovery fixture launched from the PipeSpace root.

A full-capability Skill must cover every surface currently supported by that adapter. It must not include directories whose source grammar has not been frozen or that have not been verified against a real client. Adding a capability requires corresponding positive, conflict, invalid, and E1 discovery cases.

These cases must execute the final `pipebuilder.py` through a subprocess rather than invoking an adapter class or renderer directly. A platform capability may be marked supported only when it has an implementation, offline E2E coverage, real-client E2E coverage, and a compatibility note.
